# 界面异常状态设计规范（状态池）

> **说明**：此文件定义了需要生成的 7 种异常状态的完整规范。
> 后端函数会读取此文件内容，将其作为系统提示词的一部分发送给 AI 模型，
> 以确保生成的 HTML 代码符合统一的设计标准。
>
> 本项目遵循以下完整界面状态定义，生成异常页面时需覆盖全部适用状态。

---

## 规范总则

1. **视觉一致性**：所有异常状态页面应与原始 UI 设计保持一致的配色、字体、圆角、阴影等视觉风格
2. **可访问性**：确保对比度符合 WCAG AA 标准，图标配有文字说明
3. **响应式**：异常状态组件需适配移动端和桌面端（375px ~ 1440px）
4. **交互反馈**：提供明确的操作引导（如"重试"、"返回"按钮）

---

## 异常状态类型（7 种）

### 1. 网络异常（Network Error）

**描述**：无网络或连接不稳定

**触发场景**：网络断开、请求超时、服务器无响应、DNS 解析失败

**必须包含的元素**：
- 错误图标（断网图标、警告图标）
- 明确提示文案（如"网络不可用"、"无法连接到服务器"）
- 错误详情描述（简要说明原因）
- **重试按钮**（核心操作）
- 次要操作（如"检查网络设置"、"返回首页"）
- 若可行，展示缓存内容（离线可用场景）

**技术建议**：
- 提供自动重试机制（带指数退避）
- 显示错误码或时间戳便于排查
- 区分于服务端错误的视觉表现（不同图标/颜色）

---

### 2. 空数据状态（Empty State）

**场景**：页面有结构但无数据（首次使用）、无历史记录、搜索/筛选无结果

**必须包含的元素**：
- 居中的空状态插画/图标（尺寸建议 120px ~ 160px）
- 简洁的标题文案（如"暂无数据"、"还没有内容"）
- 辅助说明文案（解释为什么是空的）
- 主要操作按钮/入口（如"去逛逛"、"新建"、"添加"、"去浏览"）
- 次要操作链接或提示（可选）

**布局要求**：
- 内容垂直居中，占据容器的主要空间
- 文字与按钮间距合理（16px ~ 24px）
- 搜索无结果场景应额外提供搜索建议或热门关键词

---

### 3. 加载中状态（Loading State）

**场景**：页面初始化、数据请求、文件上传等

**必须包含的元素**：
- 加载动画
- 加载文案（如"正在加载..."、"请稍候"）

**实现方式**（按场景选择）：
- **骨架屏 Skeleton**：首屏加载使用，与原页面布局一致，用灰色块模拟内容轮廓，体验最流畅
- **Spinner**：圆形旋转动画，适用于局部/轻量级加载
- **进度条**：适用于可预估进度的长耗时操作（如文件上传）

**注意事项**：
- 避免使用全屏遮罩阻塞用户操作（除非必要）
- 加载时间超过 3 秒应显示预计等待时间或取消选项
- 骨架屏的布局结构应与真实内容区域一一对应

---

### 4. 错误 / 失败状态（Error / Failure）

**场景**：操作失败、服务器 5xx 错误、表单提交失败、接口返回业务错误码

**必须包含的元素**：
- 错误图标（区别于网络异常，可用服务器/bug 图标）
- 友好的错误标题（如"服务器出了点问题"、"操作失败"、"系统繁忙"）
- 明确的错误描述（告诉用户出了什么问题）
- 解决方案指引（告诉用户可以做什么）
- 操作按钮："重试"或"返回"，以及反馈入口（如"联系客服"、"报告问题"）

**注意**：
- 不要暴露具体的技术错误信息（堆栈、内部 API 路径等）给终端用户
- 与网络异常在视觉上做区分（不同图标/颜色/文案语气）
- 表单类错误应在对应字段旁内联展示，而非整页替换

---

### 5. 权限 / 访问限制（Permission / Access Denied）

**场景**：无模块权限、需要登录、角色不足、资源访问被拒绝

**必须包含的元素**：
- 权限相关图标（锁图标、禁止图标）
- 标题（如"没有访问权限"、"暂无权限"、"需要登录"）
- 说明文案（解释为什么无权限，以及如何获取权限）
- 操作引导（如"申请权限"、"联系管理员"、"切换账号"、"升级账户"）
- 返回上一页或首页的导航

**设计要点**：
- 语气友好但不模糊，明确告知用户限制原因
- 如果有获取权限的路径，务必清晰展示（不要让用户卡死在这里）
- 登录态丢失的场景应优先引导重新登录

---

### 6. 内容缺失状态（Content Missing）

**场景**：内容被删除、页面不存在（404）、功能建设中、数据被过滤后无结果

**必须包含的元素**：
- 缺失状态的视觉元素（404 图形、建设中插画、已删除标记等）
- 友好标题（如"页面走丢了"、"内容不存在"、"功能开发中"）
- 辅助说明（简要解释原因）
- 导航路径（如"返回首页"、"查看其他内容"、"搜索其他内容"、"查看热门推荐"）

**特殊处理**：
- **404 页面级缺失**：可增加趣味性插画，降低用户的挫败感
- **搜索结果为空**：归入空数据状态（第 2 种），提供搜索建议或热门关键词
- **功能建设中**：可加入预计上线时间或订阅通知入口
- **内容被删除**：提供相关推荐内容的替代路径

---

### 7. 极端数据状态（Extreme Data）

**场景**：内容过载、超长文本溢出、数字溢出、大图加载缓慢、列表数据量过大

**必须包含的元素**：
- 合适的数据展示策略（根据极端类型选择）
- 用户可控的操作入口（展开/收起、分页切换、加载更多等）
- 性能相关的友好提示（如"图片较大，加载中..."）

**技术实现方式（按场景选用）**：
- **超长文本**：截断 + "展开全部"/"收起" 按钮，避免撑破布局
- **数字溢出**：使用科学计数法或简化显示（如"1.2万"而非"12345"）
- **大图/大量图片**：懒加载（Lazy Load）+ 渐进式图片加载（模糊→清晰）+ 占位符
- **列表过载**：虚拟滚动（Virtual Scroll）/ 分页 / "加载更多"
- **存储/配额超限**：使用进度条或环形图直观展示用量占比（如"已用 100MB / 共 100MB"），接近上限时（>80%）提前预警；提供升级/清理引导（如"升级套餐"、"清理空间"、"管理数据"）

## HTML 输出规范

每个异常状态生成独立的 HTML 文件，遵循以下规则：

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>[异常状态名称] - [原组件名]</title>
    <style>
        /* 内联样式，确保文件可独立运行 */
        /* 使用 CSS 变量方便后续主题定制 */
    </style>
</head>
<body>
    <!-- 异常状态内容 -->
</body>
</html>
```

**输出要求**：
- 每个 HTML 文件完全独立，不依赖外部资源（内联 CSS 和 SVG 图标）
- 文件命名格式：`[原组件名]-[异常状态英文].html`
- 包含中文注释标注各区域用途
- 响应式适配（375px 移动端 ~ 1440px 桌面端）
